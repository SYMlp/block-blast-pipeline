这是现在能跑的流水线。但它不是一开始就长这样——下面是它怎么演化成这样的。

①

演进过程:撞墙 → 改

每个节点 = 当时做了什么 · 撞到的痛点 · 怎么改的。这条线本身就是我想展示的东西。

1 · 从 JD 原文拆名词,当调研入口

没有空想"做个游戏"。把招聘 JD 原文里的技术名词逐个拆出来——SDD、Harness、休闲游戏、A/B 批量、Spec/DSL、Coding Agent、Cocos、DevOps——每个当成一条要搞懂的深度调研入口。

这样"懂不懂这个岗"变成可逐条验证的,而不是泛泛而谈。

2 · 第一版搭出来,红队一审就翻车最关键的一次

深调研手感/方块数值/A-B 维度后,搭了第一版:参数化 config + 4 个变体,还写了漂亮的说明。

痛点:开一个独立红队来审,抓出"死参数"——说明里把引擎根本没读的参数(density、DDA)当成 A/B 维度在展示。grep 一下,0 命中。面试官一眼就能戳穿。
怎么改:立了条铁律——能做成真的就做成真,做不成就删,绝不留着假装。把 density(开局预填)和 DDA(按占用率调发片)真实现进引擎,变异实验证明生效(关掉→开启,平均分 63.9→634.8)。

3 · "六个旋钮"被一句话戳破 → 三层 DSL

本来打算把变体当"六个参数旋钮"列出来。但一个更狠的问题冒出来:人类的感受(紧张、公平、爽)和程序参数之间,到底怎么转换?

痛点:这个转换跟我熟悉的 CRUD 不一样——CRUD 里"名称必填→required:true"是无损 1:1;游戏里"想要更紧张"对应一束参数,方向靠猜,而且对不对只能事后测。这正是休闲游戏公司一年跑一万次 A/B、成功率仅 3% 的根本原因。
怎么改:把变体升级成三层——意图(人话假设 + 目标代理带)/ 参数 / 验证(harness 自动比对)。DSL 的价值不是列参数,是当"翻译 + 验证"的台面。

4 · 不是头一回做 DSL——迁移自己的方法论

这套声明式 DSL 我在另一个项目(一套等保 SOC 系统)里已经跑在 50+ 个实体上:一份 model.yaml → 多个 generator → 前端/后端/DB/测试。我把那份"DSL 设计方法论"迁过来。

梳理清楚:那份文档自带一道防注水闸——"只有一个消费者 = 配置文件,不配叫 DSL"。逼自己把变体做成一源多投影(引擎 config + harness 验收 + A/B 登记),才够格。

5 · 表面找不到答案 → 认知推导到底层

卡在一个问题上:到底该测什么,才能代表"好玩"?手感技巧、A/B 维度这些表面调研都给不出答案。

怎么改:开四路认知推导挖到根上——玩的本质(Koster:fun = 大脑掌握模式的快感)→ 休闲特化(把胜任感打包成零摩擦微剂量)→ 可测边界。挖出拱顶石:难度 = 把玩家卡在「无聊崖」(模式被吃透)和「挫败崖」(模式装不起)之间的窄带。转换不再是"调大调小",是双边带。

6 · 凭直觉选的指标,被真实数据淘汰

推导完,我凭直觉押"棋盘占用率"当公平感的代理指标。

痛点:真跑 sim 一看——占用率在四个变体间是平的(0.61–0.66),被千篇一律的贪心 agent 抹平了,根本区分不开。
怎么改:淘汰它,换成实测强区分的三个:rescueRate(安全网救场率=公平)、rewardEntropy(清除规模的熵=惊喜)、stepsP90(单局长度=节奏)。指标不是猜的,是被数据筛出来的。

7 · 闭环成真:门禁有牙齿 + AI 真的会改

三层 DSL + 验证门禁 + 真 AI 生成,全部跑通(下方是真实输出)。两件事证明它不是花架子:一个变体被自动淘汰,一个变体由 AI 反复改三次才达标。

②

真实结果:门禁裁决

harness 拿从"玩的本质"推导出的指标,逐条比对每个变体的实测代理,落带内才放行。

变体裁决为什么
control 基线PROMOTE四项代理全落带内
compact 紧凑PROMOTE节奏最快、惊喜最足(rewEnt 0.56)
hard-mode 高难PROMOTE刻意贴挫败崖,但验明没跌破底线
ai-brisk 明快(AI 生成)PROMOTEAI 三次收敛产出,见下
relaxed 宽松REJECTrewEnt 0.12<0.2(太闷)、stepsP90 170>60(拖沓)→ 贴无聊崖
relaxed 被自动枪毙,才是门禁有牙齿的证据。一个"看起来人畜无害的轻松版",harness 判它跌进无聊崖、打回——不靠人肉试玩。全 PASS 的门禁等于没门禁。
③

AI 生成:一句假设 → 三次收敛

输入一句人话:"给熟练玩家的明快变体:节奏快、消除频繁、惊喜足,但保持公平,单局仍碎片化"。AI 自己定目标带、调参数,harness 把失败结构化喂回去重试。

尝试发生了什么裁决
1rescueRate 0.107 略超 0.1(稍不公平)REJECT
2过冲:修好公平(0.000),却把节奏 0.430、惊喜 0.344 双双拉出带REJECT
3打开 DDA + 收紧占用阈值对冲大片不公平 → 四项全 PASSPROMOTE
attempt 2 的过冲,正是"感受↔参数"翻译有损的活证据:修公平性牺牲了节奏。AI 拿着 harness 的结构化反馈把两者重新配平,第三次用 DDA 这个旋钮解开了矛盾。这就是"多步推理 + 反思自修复",真跑出来的,不是一把过的运气。
④

承重墙:换 payload,不换架构

整条流水线不是为这个 demo 现搭的——它和我另一个项目的声明式 DSL 是同构的,换了处理对象而已。

SOC 系统的 CRUD-DSL(已跑在 50+ 实体)游戏变体 DSL(本作)
一个 CRUD 实体→一个游戏变体
字段挂 list/form/query 多维→变体挂 board/spawn/scoring/difficulty 旋钮
一源多投影(前端/后端/DB/测试)→引擎 config + harness 验收 + A/B 登记
分层护栏防漂移→schema·sim·verify 四道门
为什么有效:两套都是"声明意图 → 多目标投影 → 校验防漂移"。同构意味着已有方法论直接迁移——这才是真正难被替代的差异点。
⑤

诚实边界:什么是真的,什么还没做

  • ✓ 真跑、可复现:纯 TS 逻辑核(奇偶校验证明与旧 JS 引擎零行为漂移)、13 项属性测试、headless 模拟门禁、三层 DSL + verify、claude 真生成的 ai-brisk。
  • ◐ 进行中:技术栈对齐 Cocos Creator 3.8 + TS——逻辑核已就绪,渲染壳在搭。
  • ○ 刻意没做 / 测不了:LLM 评估层(L3)是骨架;Codebase RAG 当前代码量小不需要;真正的"好不好玩、留存、付费"必须真玩家 A/B——headless 只是第一道粗筛闸,不冒充全能。
这道边界写在脸上,本身就是想展示的工程判断力:不吹"全自动"、不把骨架说成跑通。
一句人话假设 → AI 翻译成参数 → harness 验证那个翻译成没成 → 促或汰。
