2

拆解已有的 SOC 流程

Decompose the proven flow
做了什么
把我之前跑通的企业级 SOC / 智能派单 Agent 的状态机拆出来：告警 → 分类 → 派单 → 核验 → 失败重派，外加治理五件套（护栏 / 人工确认 / 自动评估 / 可观测 / 成本控制）。
为什么从它起
这是一个已经在真实企业场景验证过的状态机，带完整治理。不从零设计，直接拿成熟资产当地基。
这是整件事的承重墙：我不是来学做 Agent 的——我已经有一个跑通的。后面要做的，只是给它换个处理对象。
3

类比替换到游戏生成场景

Analogy-map to game variants

把 SOC 流程的每个节点，一对一替换成"生成游戏变体"的节点。同一套架构，换了处理对象：

SOC 派单（已验证）游戏变体生成（本作）
告警分类→解析 Spec（parse_spec）
自动派单→生成变体代码 / config（generate）
核验工单→跑三层 Harness（run_harness）
失败重派→失败自修复（reflect_and_fix ≤3 次）
护栏 / 人工确认 / 可观测→整套原样迁移
为什么有效：两个场景是同构的——都是"接收一个意图 → 自动产出 → 验证 → 不过就修"。同构意味着换 payload、不换架构，已有经验直接复用。
4

搭建可执行流程

Build the pipeline

把上面的设计落成真能跑的代码。一条线：

Spec(YAML) → 参数化 config → 纯JS 引擎 → 三层 Harness → 4 个 A/B 变体 → CI/CD

  • 纯 JS 引擎（零依赖、无 DOM）—— 所有玩法参数从 config 读，能在命令行 headless 跑，才能被自动验证。
  • 一张 config schema —— 棋盘 / 生成 / 计分 / 难度 / 手感全收敛成参数，每个字段就是一个 A/B 维度。
  • 4 个变体 —— control / 紧凑 / 宽松 / 高难，各是同一份 schema 的不同取值。
  • 三层 Harness —— 属性测试 + headless AI 玩家 + LLM 打分（详见下一步）。
  • CI/CD —— 每次提交自动跑门禁，不过不让合并。
为什么这样拆："参数化"是贯穿全线的脊柱——把游戏的一切可变量收敛成一份 config，流水线才能自动流动、一键喷出 N 个变体。
